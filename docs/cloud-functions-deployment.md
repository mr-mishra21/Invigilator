# Cloud Functions deployment guide

## Structure

```
backend/
├── .firebaserc          — project aliases (default = invigilator-78dc2)
├── firebase.json        — firestore rules path + functions config
└── functions/
    ├── package.json
    ├── tsconfig.json
    └── src/
        ├── index.ts             — entry point, exports all functions
        ├── onConsentCreate.ts   — Firestore trigger: stamps signedAt, validates hash
        ├── claimLinkingCode.ts  — HTTPS callable: validates + claims 6-digit code
        └── consentHashes.ts     — expected SHA-256 map, generated by sync script
```

## First-time setup

```bash
cd backend/functions
npm install
```

Requires Node 20+. Check with `node --version`.

## Build

```bash
cd backend/functions
npm run build
# Output goes to backend/functions/lib/
```

## Deploy

From the `backend/` directory:

```bash
firebase deploy --only functions
```

To deploy rules and functions together:

```bash
firebase deploy --only firestore:rules,functions
```

## Local development with emulators

```bash
cd backend
firebase emulators:start
```

Emulator UI: `http://localhost:4000`

Functions run at `http://localhost:5001`
Firestore at `http://localhost:8080`
Auth at `http://localhost:9099`

## Updating consent hashes

When you edit a consent text file or bump a version:

1. Edit `core/src/main/assets/consents/<file>.txt`
2. Bump the version in `core/src/main/kotlin/.../consent/ConsentVersion.kt`
3. Run the sync script:
   ```bash
   scripts/sync-consent-hashes.sh
   ```
4. Rebuild the Android app (the `verifyConsentHashes` Gradle task regenerates `ConsentHashes.kt`).
5. Redeploy functions:
   ```bash
   cd backend && firebase deploy --only functions
   ```

The `onConsentCreate` Cloud Function will reject any consent record with a hash that does not match the table in `consentHashes.ts`. The Gradle task on the Android side ensures the app always sends the correct hash for the current asset. Both must be in sync.

## Function details

### `onConsentCreate` (Firestore trigger)

- **Trigger:** `onCreate` on `/consents/{consentId}`
- **Does:**
  - Writes `signedAt` as a server timestamp (client must not send this field — enforced by Firestore rules)
  - Looks up the expected hash for `consentType + consentVersion + consentLanguage`
  - If hash matches: record is finalized
  - If hash mismatches or type/version/language is unknown: sets `invalidated: true` + `invalidationReason`, logs an error
- **Does not:** write `ipHash` — Firestore triggers don't see client IPs. If IP logging becomes a legal requirement, migrate consent creation to an HTTPS callable.

### `claimLinkingCode` (HTTPS callable)

- **Called by:** parent's Android app via `FirebaseFunctions.getHttpsCallable("claimLinkingCode")`
- **Input:** `{ code: string }` — the 6-digit code the parent typed
- **Validates:**
  - Caller is authenticated
  - Code is exactly 6 digits
  - Code document exists in Firestore
  - Code is not already claimed
  - Code has not expired (30-minute TTL)
  - Caller is not the student themselves
- **Transaction:** atomically marks `claimedBy` + `claimedAt` — prevents two parents racing to claim the same code
- **Returns:** `{ studentUid, studentDisplayName, studentDateOfBirthMillis }` so the parent confirm screen can show "Confirm: Aarav Mishra, age 16"
